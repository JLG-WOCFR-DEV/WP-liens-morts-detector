- **Planification maîtrisée** : les réglages exposent la fréquence, les plages de repos, les délais entre liens/lots, les tailles de batchs et les timeouts HTTP pour ajuster la charge du scanner à son hébergement.【F:liens-morts-detector-jlg/includes/blc-settings-fields.php†L35-L498】
- **Actions rapides en administration** : la liste des liens propose filtrage par type, recherche, actions en ligne (édition, dissociation, suggestion/appliquer une redirection, recheck, ignore) et traitements groupés pour gagner du temps sur la remédiation.【F:liens-morts-detector-jlg/includes/class-blc-links-list-table.php†L90-L200】【F:liens-morts-detector-jlg/includes/class-blc-links-list-table.php†L530-L881】
- **Scans liens & images** : les lots de liens s'exécutent via WP‑Cron ou manuellement, tandis que les images disposent d'une file dédiée avec verrouillage, prise en charge des médias distants et reprise en arrière-plan.【F:liens-morts-detector-jlg/includes/blc-admin-pages.php†L147-L220】【F:liens-morts-detector-jlg/includes/blc-scanner.php†L3214-L3245】【F:liens-morts-detector-jlg/includes/blc-scanner.php†L380-L515】

## Renforcements récents
- **Orchestration traçable** : les déclenchements manuels génèrent un identifiant de job, déclenchent une tentative de replanification automatique et alimentent un historique persistant (états, erreurs, horodatage).【F:liens-morts-detector-jlg/includes/blc-admin-pages.php†L149-L226】【F:liens-morts-detector-jlg/includes/blc-scanner.php†L25-L148】
- **Client HTTP durci** : les requêtes sortantes appliquent des timeouts dédiés, une rotation d’identité, un lissage de débit et un backoff tenant compte des en-têtes `Retry-After` pour limiter les faux positifs réseau.【F:liens-morts-detector-jlg/includes/Scanner/RemoteRequestClient.php†L7-L192】
- **Scanner mieux segmenté** : la préparation des lots (configuration, verrou, heuristiques soft 404) est isolée dans des méthodes dédiées, facilitant les tests ciblés et les extensions personnalisées.【F:liens-morts-detector-jlg/includes/Scanner/ScanQueue.php†L1-L180】【F:liens-morts-detector-jlg/includes/Scanner/ScanQueue.php†L260-L340】
- **Télémétrie exploitable** : chaque run publie des métriques (durée, progression, réussite) persistées côté WordPress et exposées via un hook pour alimentation de dashboards externes.【F:liens-morts-detector-jlg/includes/blc-scanner.php†L3214-L3250】

## Lacunes face aux outils professionnels
- **Pas de supervision native** : l’historique et les métriques sont stockés côté base, mais aucune interface n’expose encore ces données (tableaux ou graphiques) pour des équipes support non techniques.【F:liens-morts-detector-jlg/includes/blc-scanner.php†L25-L148】【F:liens-morts-detector-jlg/includes/blc-admin-pages.php†L147-L236】
- **Toujours mono-instance** : l’orchestration repose sur WP‑Cron local, sans file distribuée ni workers parallèles capables de répartir la charge sur plusieurs nœuds comme dans les plateformes SaaS.【F:liens-morts-detector-jlg/includes/blc-admin-pages.php†L149-L236】【F:liens-morts-detector-jlg/includes/Scanner/ScanQueue.php†L63-L120】
- **Absence de proxy/smart routing** : malgré le backoff et la rotation d’User-Agent, le client HTTP ne gère pas encore la rotation de proxys ou les pools IP, éléments clés pour contourner des limites réseau agressives.【F:liens-morts-detector-jlg/includes/Scanner/RemoteRequestClient.php†L7-L192】
- **Alerting avancé à construire** : aucun seuil dynamique ou canal dédié (Slack, PagerDuty) n’est branché sur les métriques pour escalader automatiquement en cas de dégradation prolongée.【F:liens-morts-detector-jlg/includes/blc-scanner.php†L3214-L3250】
